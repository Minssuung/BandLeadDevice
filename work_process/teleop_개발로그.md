## 2. 축 분해 방식의 진화 — 4번 갈아엎음

같은 동작(예: 굴곡40°+벌림50°)을 각 관절각으로 분해하는 방법. fact-checker 2회 검증.

| 방식 | 순수동작 | 2축결합 | 3축결합 | 치명적 문제 |
|------|---------|---------|---------|------------|
| **오일러 XYZ** | OK | — | — | **벌림 90°서 굴곡값 폭발**(40→101). 축 뒤바뀜 |
| **독립 swing-twist** | 정확 | 40,50,**-19**(헛돎) | 26,57,11 ❌ | 비직각축서 교차오염 |
| **순차 swing-twist(비직각)** | 정확 | 40,50,0 ✅ | 54,52,19 | 3축 결합 부정확 |
| **Gauss-Newton IK** | 정확 | 정확 | **30,40,50 정확** ✅ | **실물서 워밍스타트 감김 → j6=+2326°! → 모터 폭주** ❌ |
| 메모리없는 직교 swing-twist | 정확 | 정확 | 79,54,25(≈) | 감김 없음. 단 3축결합 부정확(j3/손목) |
| **swing-twist 시드 + 메모리없는 IK** ⭐ 최종 | 정확 | 정확 | **60,50,40 정확** ✅ | 없음 (감김 불가 + 결합 정확) |

> ⚠️ **헷갈리기 쉬운 점 — 지금 쓰는 것 (2026-06-10 갱신):**
> - **캘리브레이션(보정)** = **씀** ✅ (functional 축 → `calibration.json`)
> - **워밍스타트 IK**(이전 프레임 시드) = **안 씀** ❌ (감김 j6=+2326° 때문에 폐기)
> - **메모리없는 IK**(매 프레임 swing-twist 해로 재시드) = **씀** ✅ — 이게 핵심
>
> 즉 현재 = **"캘리브레이션 + swing-twist 시드 + 메모리없는 IK 다듬기"**.
> swing-twist가 **바운드된 시드**(±180)를 주고 → IK가 **결합 정확**까지 다듬음 → 둘 다 해결.
> 시드를 이전 프레임이 아니라 **매 프레임 swing-twist에서** 새로 만들기 때문에 감김이 구조적으로 불가능.
> (`decompose()` in calib_ik.py. 36µs/회. fact-checker 2차가 권고한 그 방식.)
>
> **이전 사용자 보고**: "j3(어깨비틀)·j5,6,7(손목) 부정확" = 순수 swing-twist의 3축결합 교차오염(60+50+40→80,51,23). 위 최종방식으로 해결(→60,50,40).

### 2-0. 팔꿈치(j4) 언더벤드 — 해결됨 ✅ (2026-06-10)
| | 내용 |
|---|---|
| **문제** | 팔꿈치 구부린 만큼 안 굽음. 보정축이 ~45° 어긋나면 `굽힘×cos(45°)`만 나옴(투영 손실) |
| **수치** | 보정축 45° 오차 시: 실제 90°→73°, 130°→115° (언더벤드) |
| **해결** | 팔꿈치는 1축(경첩)이라 **회전 '크기'**(2·atan2(‖qv‖,w))를 굽힘각으로, 부호만 축 투영. → **축 어긋나도 100°→100° 정확** (`hinge_angle()`) |

### 2-1. IK(반복 솔버) 폐기 사유 (fact-checker 2차)
> "이미 자세를 측정(q_rel)했는데 IK로 '찾는' 건 잘못된 도구. 워밍스타트=무한해(θ+360k) 중
> 가지 추종 → 감김/whip. 비직각축(손목 71°)=분해 애매."

- **수치**: 실물 로그서 j5 범위 **-369°~+202° (570°!)**, j6 **+2326°** → 모터 배배 꼬임 → **사용자 전원 차단(e-stop)**
- **교체**: 보정축 직교화 → atan2 순차 분해 → 매 프레임 새로(워밍스타트 삭제) → **atan2라 ±180 바운드 → 감김 불가**
- **검증**: 발산했던 케이스도 이제 **-127°로 바운드**(2326°→127°), 메모리 없음(같은 입력=같은 출력)

### 2-2. 직교화 제거 + 정확/안전 양립 (2026-06-10, fact-checker 3·4차)
| 단계 | 내용 |
|------|------|
| **교차오염 발견** | "순수 벌림인데 j1도 11° 움직임" = 보정축을 **직교화하면서 진짜 벌림축을 10° 비틂** (어깨 a·b=100°, 90°로 강제) |
| **직교화 제거** | IK 다듬기는 비직각축도 정확히 풂 → **측정축 a,b,c 그대로 사용**. 순수 굴곡/벌림/비틀 → [50,0,0]/[0,50,0]/[0,0,50] 깨끗 |
| **정확도** | 결합 60+50+40 → **60,50,40** (swing-twist만은 80,51,23로 j3 부정확했음) |
| **⚠️ 안전 재검증(4차)** | 무댐핑 IK는 특이점서 1/det 폭주 가능(2326° 재발경로) → **NO-GO** 판정 |
| **하드닝** | `decompose`: **LM 댐핑**(특이점서도 유한)+**스텝≤30°/iter**+**출력 시드±60° 클램프**+잔차수렴+최선해. → 출력 **구조적 바운드** |
| **검증** | 5만개 병적입력: 시드±60° **절대 안 넘음, 폭주 0건**. 정확도 ±70°결합 평균 **0.02°** |
| **팔꿈치 재수정** | `hinge_angle`(회전크기)는 **회내(forearm twist)를 굴곡으로 오인** → `twist_angle`(축투영)로: 회내 거름·부호有·±180바운드. (단 풀벤드엔 좋은 팔꿈치 보정 필요) |
| **추가 안전** | `_canon`(쿼터니언 w≥0, 더블커버 부호점프 차단) · **프레임당 Δθ≤12°** 명령단 클램프 · IMU 끊김 0.4s면 **freeze** · 보정축<30° 경고 |

> **현재 분해 = "측정축 직접 + swing-twist 시드 + LM 댐핑 IK"**: 정확(0.02°) + 구조적 바운드(시드±60°) + 감김 불가.
> fact-checker 4차: **토크OFF 드라이런 GO** (토크ON 전엔 Davenport 폐형해/데이터구동 게이트 권장).

### 2-3. 5차 fact-checker 감사 — 앱층 구멍 6개 수정 (2026-06-10)
분해층은 PASS(바운드 증명 전 경로 확인). 앱층에서 발견된 진짜 버그들:
| 발견 | 수치 | 수정 |
|------|------|------|
| 타이머 20Hz ≠ 패킷 5-6Hz | 같은 패킷 3-4회 재처리 → LPF 실효α 0.4→**0.78-0.87**(무력화), 클램프 12°/frame=**240°/s**(모터 54°/s의 4.4배=무의미) | **새 패킷만 처리** + **벽시계 속도클램프 40°/s** |
| **NaN 구멍** | NaN이 `max(lo,min(hi,nan))`을 뚫음 → **풀스케일(120°) 명령!** | 파싱단 가드(finite+크기) + decompose 최종 가드 |
| 클램프 로그 플러딩 | 폴트 중 최대 140줄/s → GUI 멈춤(폴트 증폭) | 진입 1회+1초 요약+해제 로그 |
| stale 복귀 런지 | 끊긴 새 팔 이동 → 복귀 순간 풀속도 추격 | **램프인 8°/s × 2s** |
| 기준설정(재0점) 런지 | prev/LPF/모터베이스 안 리셋 → 재설정 순간 점프 | 기준설정 시 전부 리셋+베이스 재캡처 |
| w=0 더블커버 모서리 | 정확히 180° 회전서 q/-q 불일치 | `_canon` 타이브레이크 |
검증: NaN/0 쿼터니언 → [0,0,0], w=0 q/-q 동일, 2만개 바운드 유지, 60+50+40 정확 유지.
**드라이런 GO. 토크ON 게이트: 팔꿈치 재보정(축오차 φ<20°, 로그 el_mag−j4 차이로 실측).**

## 4. 실물 로봇 안전 (fact-checker 1차: NO-GO 판정)

| 위험 | 수치/내용 | 해결 |
|------|----------|------|
| **속도 무제한** | Profile Velocity=0 = **전속력 슬램** | → **40 (~9 RPM 저속)** + Acceleration 20 |
| **관절 폭주** | IK 발산 시 수천도 명령 | → **JOINT_LIMIT_DEG 클램프 ±90~120°** (백스톱) |
| **시작 점프** | 토크 ON 시 옛 목표값으로 휙 이동 | → 토크 켜기 전 **goal=현재위치** (제자리 유지) |
| **크래시 시 토크 유지** | | → 연결해제/종료 시 **명시적 토크 OFF** |

이후 ① IK→swing-twist 교체(감김 불가) ② 관절 클램프 추가로 이중 차단.

## 5. 시뮬레이션 / 시각화 환경 (핵심 역할)

> **"실물 모터 부수기 전에 안전하게 검증하는 환경."** 위의 짐벌락·축분해·캘리브레이션
> 작업은 거의 다 여기서 만들고 검증한 뒤 실물 앱에 이식했다.

### 5-1. MuJoCo 시뮬 (구축 후 폐기)
- OpenArm 공식 모델(`enactic/openarm_mujoco`) 사용. 단일팔 → 양팔(bimanual) 전환, 오른팔만 텔레옵.
- `sim/openarm_sim.py`: UDP로 관절각 받아 qpos 주입(순기구학). 관절점검·데모 모드.
- **폐기 사유**: 무겁고(메시 30MB) 셋업 번거로움 → three.js 경량 웹툴로 전환. `sim/` 삭제됨.

### 5-2. three.js 디버그/캘리 툴 (`viz/`) — 메인 검증 환경 ⭐
의존성 없음(파이썬 stdlib + three.js CDN).
- `viz/bridge.py`: ESP UDP(4210) 수신 → 브라우저로 SSE 전달 + 정적파일 서빙. 실행 `python3 viz/bridge.py` → http://localhost:8000
- `viz/index.html`: **실제 OpenArm URDF 로봇**(urdf-loader, v2.0 양팔)을 브라우저에 띄우고 오른팔 7축을 IMU로 구동.

**이 툴에서 한 것 (실물 가기 전 전부 여기서 검증):**
| 기능 | 용도 |
|------|------|
| 센서Q ‖q‖ 실시간 표시 | 쿼터니언 레지스터 0x52 가짜(0.9)/0x51 진짜(1.0) **검증** |
| 소스 토글 (오일러↔센서Q) | **짐벌락 전/후 비교** (오일러는 수직서 튐, 쿼터니언은 부드러움) |
| 캘리브레이션 UI (4초 동작 녹화) | **functional 축 보정** + localStorage 저장 + JSON 내보내기 |
| 관절 점검 / 단독 테스트 | joint1~7 하나씩 흔들어 축·방향 확인 |
| 정렬후 각도 표 + 노란불 강조 | 어느 동작=어느 축인지 **데이터로** 확인(추측 제거) |
| 부호 토글 | 방향 맞추기 |
| 수신율(Hz) 표시 | 통신 지연 확인 |

→ **보정값(`calibration.json`)을 viz에서 만들어 실물 앱에 이식** = 시뮬↔실물 연결 고리.

### 5-3. 시뮬 ↔ 실물 검증 흐름
```
viz(시뮬)서 보정+분해 검증 → calibration.json 내보내기 → 실물 앱에 이식
                                                    ↓
                                      드라이런(토크 OFF, /tmp/bandlead_ik.log)
                                      → 명령값만 확인(모터 안 움직임) → 안전 검증
                                                    ↓
                                            모터 1개씩 저속 → 전체
```
**드라이런**: 토크 OFF면 계산·표시만 하고 모터에 전송 안 함 → 실물에서도 안전하게 명령값 검증.

## 7-1. 웨어러블 장착 변동 보완 — 2-포즈 재정렬 (2026-06-10, fact-checker GO-WITH-CHANGES)

| | 내용 |
|---|---|
| **문제** | 착용할 때마다 밴드가 조금씩 돌아감 → 보정축(센서 프레임 기준)이 어긋남. 풀 재보정(6동작×4초)은 번거로움 |
| **원리** | 같은 신체자세 2개(P1=차렷, P2=앞으로나란히·손바닥아래)에서 **중력 방향(센서 프레임)** 을 앵커(보정 직후) vs 현재 비교 → 장착변화 회전 C를 TRIAD로 계산 → 모든 보정축에 적용. **중력 비교는 바라보는 방향/지자기 yaw에 불변** |
| **수치** | 시뮬: 18° 밴드 틀어짐 + 3° 자세재현오차 → **축 오차 평균 2.8°로 복구**. 예상 실사용 정확도 ~5-8°(자세 재현성 의존) |
| **안전 게이트** | 분리각<30° 거부 · 자세재현실패(각도보존 8°) 거부 · <3° no-op(노이즈 주입 방지) · >45° 거부 · 잔차>10° 거부 · **항상 원본 앵커→원본 축**(세션 누적 금지) · 릴레이 정지 중에만 · 캡처 중 움직임(산포>3°) 거부 |
| **사용법** | 최초 1회(보정 직후): P1→P2→[적용]→앵커 저장. 이후 매 착용: P1(1.5초)→P2(1.5초)→[적용] = **~5초** |
| **기각된 대안** | 무감독 3-DOF 온라인 자동보정(ill-posed, fact-checker 동의 하 폐기) · "기준설정만 다시"(축 방향은 못 고침: 10° 축 슬립 = 굴곡의 17%가 비틀림으로 누수) |
| **보완 팁** | 밴드에 정렬 마크(물리적) — 슬립 ~5° 이내면 재정렬도 불필요 |

## 8. 남은 과제 (우선순위)
1. **드라이런 검증**: swing-twist 교체 후 로그에 감김 없는지 (토크 OFF)
2. **부호/매핑 최종 확정** (관절별 방향)
3. **팔꿈치 재보정** (어깨 고정하고 팔꿈치만 — 축 어긋남 ~45° 잡기)
4. **추가 안전장치**: 전류제한(back-drivable), 슬루레이트클램프, 워치독(데이터끊김), 물리 e-stop
5. **통신 속도 복원** (6Hz → 15Hz)
6. (장기) 부호/매핑을 캘리브레이션 단계에서 자동 결정 (수동 튜닝 제거)
