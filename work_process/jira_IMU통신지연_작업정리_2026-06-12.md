## 2. 추가 조사로 발견·해결한 지연 원인 (금일)

| 원인 | 해결 | 효과 (실측) |
|---|---|---|
| IMU 내부 칼만필터 대역폭 20Hz(기본) — 센서 자체 평활화 지연 | 3개 센서 모두 98Hz로 설정 (Modbus 레지스터 0x1F, 리드백 검증) | 센서단 지연 ~수십ms ↓ |
| **PC 와이파이 절전모드** — AP가 패킷을 ~200ms 단위로 몰아서 전달 ("툭툭 끊김"의 진범) | `nmcli powersave 2` 영구 비활성 | 끊김 이벤트 7.8건/초 → 0.33건/초 (**~24배**), 도착간격 중앙값 0ms(뭉텅이)→31ms(균일) |
| ESP 와이파이 modem sleep | `WiFi.setSleep(false)` | 송신측 지터 제거 |
| PC 릴레이 전송주기 50ms(20Hz) — 평균 +25ms | 10ms(100Hz)로 단축 (fact-checker 검증, 동결→런지 버그 1건 발견·수정 포함) | 평균 대기 25→5ms, 지연 분산 제거 |
| 고정 LPF(α=0.6) 필터 지연 21ms 고정 | One Euro 필터로 교체 (속도 적응형) | 빠른 동작 시 21ms→8~18ms, 정지 평활은 오히려 개선 |

부수 성과: 진단 도구 `imu_gap_probe.py` 신규 작성 (도착간격 vs ESP 송신간격 대조로 ESP멈칫/네트워크/배선/센서 원인 자동 판별).

## 3. 종단 지연 정량 검증 (폰 슬로모 240fps, 모션 상호상관 분석)

| | 개선 전 | 개선 후 |
|---|---|---|
| 사람→로봇 추종 지연 (중앙값) | ~130ms | **113ms** |
| 범위 (고신뢰 측정창) | 103~175ms | **88~125ms** |

※ 큰 휘두름 시에는 ~260ms+ 관찰 — 이는 통신 지연이 아니라 안전 속도제한(180°/s) 포화에 의한 동적 지연으로 별도 항목.

## 4. 주의사항 (재발 방지)

- **WT901C-485 RRATE 레지스터(0x03)를 200Hz 등으로 설정 금지**: Modbus 배선에서도 능동 스트리밍이 켜져 버스 플러딩(폴링 95%+ 실패) 발생 — 금일 실험으로 확인 후 0x0D(출력 안 함)로 복구·고정. 레지스터 갱신율은 RRATE와 무관하게 내부 융합 속도를 따름.

## 5. 추가 발견 — 오일러 각도 스케일 버그 (수정 완료, 재플래시 대기)

쿼터니언→오일러 역산 검증 중 발견: WT901C485 Modbus 각도 레지스터 스케일은 raw/32768×180인데 기존 펌웨어가 raw/100 사용 → **전송된 roll/pitch/yaw가 일관되게 1.82배 과장**돼 있었음 (실기 쿼터니언·오일러 쌍 6개 값 대조, 0.02° 이내 일치로 확정). 쿼터니언 기반 IK 텔레옵 경로는 무영향(그래서 그동안 정상 동작). 비-IK(rpy) 모드와 화면 표시값은 수정 후 정상 스케일로 변경됨.

## 6. 최종 적용 완료 (당일 마감)

1. **쿼터니언-온리 폴링 (응답 53B→13B) — 적용·실측 완료**: 수신율 **32Hz → 54Hz** (qo_fail=0, poll_fail=0, 3센서 전부). 90Hz 미달 사유는 센서의 Modbus 응답 자체 지연(~3-4ms/요청, 센서 펌웨어 특성). fact-checker 감사·수정 포함(타임아웃 캐스케이드 방지, 무효 쿼터니언 가드, 센서 백오프/재프로브).
2. **명령-실위치 괴리 워치독 — 적용 완료**: 0.5초 주기 모터 순회 read, 지속적 30°+ 괴리(충돌/걸림) 시 전달 자동 freeze. 오픈루프 제어의 안전 보강.
3. 기각된 방안(fact-checker NO-GO): MAX_VEL 200°/s 상향(모터 한계 마진 3%), 모터 가속 2배(슬랩 에너지 2배) — 안전상 보류.
4. 별도 관찰 항목: 센서/ESP 간헐 전원 이상 재발(당일 1회, 전원 사이클로 해소) — 커넥터/전원라인 점검 필요.
