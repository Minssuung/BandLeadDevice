# IMU 통신 지연 개선 보고 (2026-06-12)

목표: 사람 움직임 → 로봇 반응을 "거의 동기화" 느낌으로. 이날 작업은 **전달 경로의 지연·끊김 제거** 1단계.

## 오늘 적용한 변경

| # | 변경 | 위치 | 효과 |
|---|---|---|---|
| 1 | IMU 내부 필터 대역폭 20Hz → **98Hz** (3개 센서 모두, 리드백 확인) | 펌웨어 `configureImusLowLatency()` | 센서 자체 평활화 지연 단축 (추정 ~30-40ms ↓) |
| 2 | ESP 와이파이 절전 OFF (`WiFi.setSleep(false)`) | 펌웨어 `connectWiFi()` | 송신측 지터 제거 |
| 3 | **PC 와이파이 절전 OFF** (`nmcli … powersave 2`, 영구) | PC (PinkLAB 연결) | **"툭툭 끊김"의 진범 해결** — 아래 실측 |
| 4 | RRATE=출력안함 고정 + 부팅 보레이트 경합 수정 | 펌웨어 | 폴링 무결성 (poll_fail 0), 32Hz 안정화 |

※ 사고/복구: RRATE 200Hz 실험이 능동 스트리밍을 켜서 버스 플러딩 발생 → 재시도 쓰기로 완전 복구. **RRATE는 절대 만지지 말 것** (레지스터 갱신은 RRATE와 무관).

## 실측 비교 — 패킷 도착 품질 (`imu_gap_probe.py`)

정지 상태, ESP는 양쪽 모두 ~31ms 균일 송신 확인됨.

| 지표 | 개선 전 (PC 절전 ON) | 개선 후 (절전 OFF) | 비고 |
|---|---|---|---|
| 끊김 이벤트 (공백>80ms) | **94건 / 12초** (7.8건/초) | **5건 / 15초** (0.33건/초) | **~24배 감소** |
| 도착간격 중앙값 p50 | 0ms (뭉텅이 도착) | **31ms (송신간격과 일치)** | 균일 스트림 복원 |
| p95 | 104ms | 71ms | |
| p99 | 202ms | 121ms | |
| 최대 공백 | 205ms | 141ms | 잔여는 WiFi 재전송 특성 |

해석: 개선 전엔 패킷이 ~200ms마다 5-6개씩 몰아서 도착(AP가 절전 클라이언트용으로 버퍼링) → 평균 ~100ms의 추가 지연 + 화면/로봇 끊김. 개선 후 도착 리듬이 송신 리듬과 일치.

## 진짜 종단 지연을 숫자로 재는 법 (다음에)

1. **지그 방식**: IMU를 리더암에 부착 → `compare_imu_leader.py` 캡처 → IMU 응답과 리더 엔코더의 시간 오프셋(상호상관). 직전 실측 ≤40ms (단, 그건 IMU→PC 수신까지; 모터 출력까지 포함하려면 팔로워 엔코더 동시 기록 필요).
2. **간이 방식**: 폰 슬로모(240fps)로 손과 로봇을 한 화면에 찍고 반전 시점 프레임 차 × 4.2ms.

## 2단계 (같은 날 저녁, fact-checker 검증 후 적용)

Fact-checker 판정과 조치:

| 항목 | 판정 | 조치 |
|---|---|---|
| A. 릴레이 틱 50→10ms | GO-WITH-CHANGES | **적용** + FC가 발견한 치명 버그 수정: 10ms 틱에서 램프인(15°/s) 시 슬루 2틱 < 민스텝 3틱 → 전송 동결 → 램프 종료 순간 누적분 풀속(184°/s) 런지. 민스텝을 `min(RELAY_MIN_COMMAND_STEP, _step_ticks)`로 연동해 동결 불가능하게. GUI 갱신은 5틱당 1회(20Hz) 스로틀, PreciseTimer 적용 |
| B. MAX_VEL 180→200 | **NO-GO** | 기각 — 모터 한계(206°/s) 마진 3%뿐. 괴리 누적 시 회수 1.9s, 급정지 팔로스루(오픈루프라 감지 불가). 180 유지 |
| C. LPF→One Euro | GO-WITH-CHANGES | **적용** — 패킷 도메인 동일 위치, dt=패킷간격, 기준설정 시 리셋, NaN 가드. beta 실측 튜닝(0.01→0.05): 정지 진폭 0.215°(구 LPF 0.3°↑보다 평활), 지연 100°/s=18ms·200°/s=11ms·300°/s=8ms (구 LPF 고정 21ms) |
| D. 모터 가속 80→150 | **NO-GO** | 기각 — 150=3219°/s²=팁 2.9g 슬랩(80의 2배). 80에서 추종부족 증거 없음. 80 유지 |
| E. 쿼터니언-온리 폴링 | 보류 | PC 파서가 roll 키 없으면 전 패킷 드롭 → ESP에서 quat→euler 역산 유지 필요. 다음 단계 |

FC 추가 발견 반영: 전달 중 토크 OFF→ON 시 `_relay_prev_angle` 잔존으로 풀속 추격하던 잠복 버그 → 재인에이블 시 '현재=0' 재시드 + 램프인 1초. 미반영 권고(향후): 명령-실위치 괴리 워치독(2~5Hz 저빈도 read, 오픈루프 보완).

## 종단 지연 실측 (폰 슬로모 240fps, 2단계 적용 전 기준값)

IMG_4967.mov (720p 슬로모, 8배 감속 베이크) — 좌(로봇)/우(사람 팔) ROI 모션 에너지 상호상관, 우측 끝 리더암 제외. 분석 창 400프레임(실제 1.7초), 피크상관 0.6 이상만 채택.

| 동작 유형 | 측정 지연 | 신뢰도 |
|---|---|---|
| 작은~중간 동작 (후반 구간) | **103 / 114 / 132 / 175ms** (대표 ~130ms) | 상관 0.80~0.99 (높음) |
| 큰 휘두름 (초중반 구간) | 262~292ms | 상관 0.61~0.73 (중간) |

- 기본 지연 ~130ms는 지연 예산 추정(100-160ms)과 일치.
- 큰 휘두름의 +130ms는 통신이 아니라 **속도제한 포화**(MAX_VEL 180°/s < 사람 피크속도) — 동작 진폭에 비례하는 동적 지연. 2단계로는 거의 안 줄고, 폴링 고속화/모터 상향 영역.

### 2단계 적용 후 재측정 (IMG_4969.mov, 동일 방법)

| | 적용 전 (IMG_4967) | 적용 후 (IMG_4969) |
|---|---|---|
| 기본 추종 지연 중앙값 | ~130ms | **113ms** |
| 범위 (고신뢰 창) | 103~175ms | **88~125ms** |
| 최저 | 103ms | **88ms** |

- 중앙값 ~15ms↓, 상한이 175→125ms로 크게 조여짐(틱 지연 분산 제거 효과). 예상치(90-110ms)와 부합.
- 초반 1개 창에서 439ms(상관 0.65, 임계 미달로 기각) — 전달 시작 직후 램프인(15°/s) 구간으로 추정. 정상 동작.
- 남은 ~110ms 내역(추정): 센서 ~10 + 샘플링 16 + WiFi 3 + 필터 15-25 + 틱 5 + 슬루·모터 응답 30-50.

### 3차 측정 — 쿼터니언-온리 54Hz (IMG_4970.mov, 동일 방법)

| | 2단계 전 (IMG_4967, 32Hz) | 2단계 후 (IMG_4969, 32Hz) | 쿼터니언 후 (IMG_4970, 54Hz) |
|---|---|---|---|
| 기본 추종 지연 중앙값 | ~130ms | 113ms | ~129ms |
| 고신뢰 창 범위 | 103~175ms | 88~125ms | 117~148ms |
| 큰 휘두름(속도제한) | 262~292ms | — | 332~365ms |

**해석 — 54Hz의 추가 개선은 이 측정으로 분리 불가 (예상된 결과):**
- IMG_4969(113)와 IMG_4970(129)은 슬로모 상호상관의 정밀도(±15~20ms, 영상별 동작속도·카메라거리·각도 차이) 안에서 **사실상 동일**. 측정 노이즈로 이번이 약간 높게 나왔을 뿐.
- 32→54Hz가 종단지연에 주는 건 **샘플링 단계 ~7ms**뿐 — 측정 노이즈보다 작아 분리 불가. 처음부터 예상된 한계.
- **54Hz의 실제 가치는 지연이 아니라 품질**: 패킷이 촘촘해져 빠른 동작 중 끊김/계단이 부드러워짐(상호상관 지연으로 안 잡힘). 큰 휘두름(332~365ms)은 속도제한 영역이라 폴링속도 무관.

**결론**: 통신 지연은 2단계에서 **~110ms로 바닥**. 그 아래는 슬로모로 측정 불가 영역. 남은 ~110ms 대부분은 통신이 아니라 모터 응답(프로파일)+필터 — FC가 안전상 NO-GO 준 영역이라 보류. **슬로모 측정은 여기서 졸업.**
