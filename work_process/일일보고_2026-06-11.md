## 3. 센서 융합 — 미해결 증상의 근본 규명과 모드 확정 ⭐

**여정**: 손목 +90° 오프셋 발견 → 가지점프 가설 → FK잔차로 기각 → **지자기 상대 yaw 드리프트 확정**
(오프셋이 93~129° '배회' + FK잔차 0 = 자세는 정확, 센서간 상대 방위가 표류)

**6축 전환 실험** (fact-checker GO-WITH-CHANGES, 공식 SDK로 레지스터 검증):
- 손목 단독 ≤131°/s: 오프셋 ≤3° (거의 완벽)
- 그러나 빠른 팔 휘두름: **yaw 영구 킥** (+40°/이벤트), 지속 동작 시 ~27°/분 표류

**사람 착용 A/B (최종 결정)**:

| 채널 | 6축 | 9축 |
|------|-----|-----|
| 윗팔 | 최대 166° | **13°, +1.0°/분** |
| 팔꿈치 | 133° | **4.1°, -0.5°/분** |
| 손목 | 124°, +29°/분 | **21.6°, -0.3°/분** |

→ **사람 밴드 = 9축** (자석에서 멀어 자기치유 작동, 누적 0) / **지그 검증 = 6축+중속** (자석 옆).
펌웨어 `IMU_USE_6AXIS` 토글로 1글자 전환. **"손목/ID7 가끔 이상" 장기 미제 사건 종결.**

## 4. 통신 — 한 줄의 기적

- 병목: `sendImuPacket()`이 JSON(~400자)을 **매 패킷 Serial.println** → 115200에서 패킷당 ~35ms 블로킹
- 그동안의 9~14Hz, poll_fail 수천, "센서 멈춤", 130ms 지연이 **전부 이 한 줄** (타 세션 제보로 확인)
- 수정: println 2초 스로틀 + 통합읽기 카운터(comb_ok/fail) + 적응형 스킵
- 결과: **30.8Hz, poll_fail 0, comb_fail 0** (통합읽기는 처음부터 무결했음)

## 5. 안전/품질 추가분

- **팔꿈치 충돌벽**: 모터 EEPROM 위치한계 800틱(실측) — 앱·기준자세와 무관한 영구 벽 + 소프트클램프 140°
- **위치 도메인 슬루**: 베이스 캡처가 틀려도 모터 슬램 불가 (각도클램프의 사각 보완)
- 베이스 캡처 로그(범인 추적용), 드리프트 워치독(°/분 흐름 경고), 센서 재부팅 감지(60°+ 점프 경고)
- 속도 상향: PV 150(206°/s), 추종 180°/s — 안전장치 위에서 해방

## 버그/교훈 (재발 방지)

1. **각도 비교에 선형 통계 금지** — ±180 랩+오프셋에 상관계수가 완전히 속음 → 원형 통계 필수
2. **네트워크 버스트**: PC 도착시각으로 속도 계산하면 가짜 스파이크 → ESP 타임스탬프 사용
3. 핫루프 Serial 출력은 통신 전체를 죽인다 (35ms/패킷)
4. "센서가 안 돈다" 진단 전에 **데이터 멈춤(stale)부터 검사** (IMU2 오진 소동)
5. 측정 시 "잡은 손은 바위, 움직이는 손은 달팽이" — 비고정 링크 출렁임이 축 추정 오염
6. 분해 순서는 물리 체인 순서대로 (손목 j5→j6→j7)
